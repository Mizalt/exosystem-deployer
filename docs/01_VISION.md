# 01 — Видение продукта

## Главная цель

Продукт, который **устанавливается на любой сервер одной командой** и позволяет
разворачивать приложения **через UI**, без знаний DevOps.

Финальная формула:
> Дёшёвый VPS + одна команда установки + веб-панель = работающее приложение
> с доменом и HTTPS, которое само себя поддерживает запущенным.

## Кому и зачем (бизнес-модель)

- **Аудитория:** малый и мало-средний бизнес, инди-разработчики, агентства.
- **Боль:** деплой дорого и сложно — нужен DevOps или дорогая managed-платформа
  (Heroku/Vercel/managed k8s).
- **Наше решение:** self-hosted ядро инфраструктуры. Клиент платит только за свой
  сервер; продукт берёт на себя оркестрацию, проксирование, SSL, мониторинг.
- **Монетизация (вектор, не сейчас):** Free (MVP, 1 админ) → Pro (роли, команды,
  protected mode) → Enterprise (кластер, гранулярные права).

## Три столпа (в порядке приоритета для MVP)

1. **Безопасность.** Продукт ставится на машины, торчащие в интернет. Аутентификация
   панели, защищённое хранение секретов, изоляция приложений, отсутствие открытых
   дыр «из коробки» (catchall на 403, а не открытая панель).
2. **Надёжность.** Ядро по парадигме Kubernetes: знает желаемое состояние и
   удерживает его. Падение приложения не должно вызывать лавину или каскад.
   Операции по возможности атомарны (проверка `nginx -t` перед reload, откат при
   неудачном build).
3. **Простота (для MVP).** Один администратор, один сервер, минимум понятий в UI:
   «залил → выбрал домен → работает». Сложные фичи (роли, кластер) — позже.

## Парадигма: «Kubernetes для бедных»

Ключевая идея — **декларативное управление состоянием**:

- Пользователь описывает **желаемое** состояние: «приложение X версии v1.2,
  3 реплики, домен app.example.com».
- **Оркестратор** (reconcile-цикл) непрерывно приводит **действительное**
  состояние к желаемому: запускает/останавливает контейнеры, чинит упавшие,
  обновляет конфиги прокси.
- Человек не управляет контейнерами вручную — он управляет **намерением**.

Это уже частично реализовано (`Deployment.target_replicas` + `orchestrator.py`),
и это правильное направление. Нужно довести до надёжности.

## Целевой UX установки (north star)

```bash
# Linux
curl -fsSL https://get.clouddeployer.io | sh

# Windows (PowerShell)
irm https://get.clouddeployer.io/install.ps1 | iex
```

После установки:
1. Поднимается стек в контейнерах (deployer + nginx + certbot).
2. Печатается URL панели и одноразовый пароль администратора.
3. Админ заходит в UI, заливает приложение, привязывает домен — готово.

## Принцип «всё в контейнере»

Включая сам деплоер. Стек целиком описан одним `docker-compose`, деплоер общается
с Docker через смонтированный сокет. Это даёт:
- одинаковую установку на любой машине с Docker;
- чистое удаление (снёс стек — ничего не осталось на хосте);
- соответствие принципу «один артефакт — вся система».

## Целевые платформы

Решение команды: **Linux и Windows**.
Прагматичная последовательность: **Linux-first для MVP** (там нативный Docker,
`docker.sock`, certbot — меньше edge-кейсов), затем паритет под Windows через
слой абстракции окружения. Разработка ведётся на Windows 11 (Docker + WSL).

## Что НЕ входит в продукт (анти-цели)

- Не пытаемся заменить полноценный Kubernetes для крупного enterprise.
- Не мульти-облако и не автоскейлинг по нагрузке на старте.
- Не CI-сервер (мы про деплой готового артефакта, не про сборку пайплайнов).
- MVP не про мультитенантность — один администратор владеет сервером.
