# 07 — Рунбук тестового деплоя MVP

> Пошаговый сценарий: от чистого сервера до работающего приложения с HTTPS.
> Целевая платформа — Linux (Debian 11+). Для Windows-разработки см. примечания.

## 0. Что понадобится

- VPS/сервер с **Debian 11+** (или другой Linux с Docker).
- **Docker Engine** + плагин `docker compose`, утилита `git`.
- Открытые порты **80** и **443** (HTTP/HTTPS).
- Для боевого SSL — **домен**, чья A-запись указывает на IP сервера.

Проверка готовности:
```sh
docker --version && docker compose version && git --version
```

## 1. Установка одной командой

```sh
curl -fsSL https://raw.githubusercontent.com/Mizalt/exosystem-deployer/main/install.sh | sh
```

Скрипт: проверит Docker, склонирует код в `/opt/exosystem-deployer`, сгенерирует
`DEPLOYER_SECRET_KEY` в `.env`, поднимет стек (`docker compose up -d --build`) и
напечатает **одноразовый пароль администратора**. Запишите его.

> Альтернатива (из исходников): `git clone … && cp .env.example .env && docker compose up -d --build`.

### Опционально перед стартом: контактный email для SSL

Чтобы Let's Encrypt слал уведомления об истечении сертификатов, задайте в `.env`:
```sh
DEPLOYER_ACME_EMAIL=admin@ваш-домен.tld
```
Без него выпуск идёт с флагом `--register-unsafely-without-email` (работает, но без
уведомлений).

## 2. Первый вход в панель

1. Откройте `http://<IP-сервера>/` (или домен, если уже привязан).
2. Войдите как `admin` с одноразовым паролем из лога установки.
   Пароль можно посмотреть снова: `docker compose logs deployer | grep -i password`.
3. Смените пароль / задайте домен панели в настройках.

## 3. Привязка домена панели и SSL

1. В UI укажите **домен панели** (например `admin.example.com`).
2. Убедитесь, что A-запись домена указывает на сервер (в UI есть DNS-чекер).
3. Запустите **выпуск SSL** — деплоер через certbot получит сертификат
   (webroot-челлендж по `/.well-known/acme-challenge/`) и переключит nginx на HTTPS.
4. Откройте `https://admin.example.com` — панель отдаётся по HTTPS.

## 4. Первый деплой приложения

1. **Загрузите артефакт:** ZIP с приложением (внутри — код; если есть
   `requirements.txt`, зависимости установятся). По умолчанию приложение должно
   слушать порт **80** внутри контейнера (образ собирается с
   `uvicorn main:app --host 0.0.0.0 --port 80`; свой Dockerfile — на Этапе 2).
2. **Создайте Deployment:** выберите версию (artifact), группу портов и
   `target_replicas` (сколько реплик держать). Ядро поднимет контейнеры и будет
   держать их живыми (упавшие перезапустит, после 3 неудач — `failed`).
3. **Создайте Application:** имя + домен (+ опц. SSL-сертификат). Деплоер
   сгенерирует nginx-конфиг, проксирующий домен на сервис.
4. Откройте домен приложения — запрос идёт `nginx → deployer → живая реплика`.

## 5. Проверка работоспособности

```sh
# Все сервисы стека подняты
docker compose ps

# Логи ядра (старт, reconcile, ошибки)
docker compose logs -f deployer

# Управляемые приложения-контейнеры
docker ps --filter "label=manager=cloud-deployer"
```

Локальная проверка панели без внешнего DNS (подставив Host):
```sh
curl -sk -H "Host: admin.example.com" https://<IP-сервера>/
```

## 6. Траблшутинг

| Симптом | Что проверить |
|---------|---------------|
| Панель отдаёт 403 | Домен не привязан — catchall ловит неизвестные хосты. Задайте домен панели. |
| Выпуск SSL падает | A-запись домена ≠ IP сервера; порт 80 закрыт фаерволом; rate-limit Let's Encrypt. |
| Приложение 502/503 | Нет онлайн-реплик (`docker compose logs deployer`); приложение не слушает порт 80 внутри контейнера. |
| Деплой в `failed` | Сборка/старт падает 3 раза подряд (CrashLoopBackOff). Смотрите логи контейнера приложения. |
| Порт 80/443 занят | На хосте уже есть веб-сервер. Освободите порты или остановите конфликтующую службу. |

## 7. Обновление и удаление

```sh
# Обновить код и пересобрать
cd /opt/exosystem-deployer && git pull && docker compose up -d --build

# Полностью снести (вместе с состоянием — необратимо!)
docker compose down
rm -rf data uploads nginx_configs ssl_certs acme_challenge
```

## Примечания по Windows (разработка)

- Деплоер работает и под Docker Desktop, но целевая платформа — Linux.
- Платформо-зависимые точки (путь к docker-сокету, ACME) изолированы в
  [`../app/environment.py`](../app/environment.py) — на Windows сокет резолвится в
  named pipe автоматически (см. ADR-006).
- На машине разработки порты 80/443 может занимать сторонний веб-сервер —
  освободите их для чистого локального теста.
